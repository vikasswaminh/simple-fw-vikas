QuickFW Administrator Guide

Complete guide for system administrators covering installation, configuration, routing protocols, and maintenance of the QuickFW firewall appliance.

Overview

QuickFW is a production-ready L3/L4 stateful firewall appliance built on Debian Linux with nftables, FRR routing suite, and a modern Rust-based API stack.

System Architecture

  • Base OS — Debian 12 (Bookworm) Live System
  • Firewall Engine — nftables with NFQUEUE integration
  • Routing — FRR (Free Range Routing) with OSPF/BGP support
  • DHCP/DNS — dnsmasq
  • Web/API — Rust (Axum) with TLS
  • CLI — Cisco IOS-style interactive shell

ISO Installation

Building the ISO

QuickFW uses Docker and Debian live-build for creating bootable ISO images:

# Clone the repository
git clone https://github.com/quickfw/quickfw.git
cd quickfw

# Build the ISO (requires Docker)
bash build.sh

# Output will be in output/quickfw.iso
# Build time: 10-30 minutes depending on system

Creating Bootable Media

Linux/macOS

# Identify USB device (be careful!)
lsblk

# Write ISO to USB (replace sdX with your device)
sudo dd if=output/quickfw.iso of=/dev/sdX bs=4M status=progress

# Sync to ensure complete write
sync

Windows

Use Rufus (https://rufus.ie) or Etcher (https://www.balena.io/etcher) to write the ISO to USB.

Data Destruction Warning

The dd command will destroy all data on the target device. Double-check the device path before executing.

First Boot Setup

Initial Boot Process

  1. Boot from the USB drive or ISO
  2. The system will load the Debian live environment
  3. QuickFW services will start automatically
  4. The setup wizard (TUI) will launch on the console

Setup Wizard Steps

1. Network Interface Selection

Assign roles to detected interfaces:

  • WAN Interface — Connected to upstream internet
  • LAN Interface — Connected to internal network

2. WAN Configuration

Choose addressing mode:

  • DHCP — Automatic address assignment (default)
  • Static — Manual IP, gateway, and DNS configuration

3. LAN Configuration

Configure the internal network:

  • LAN IP address (default: 192.168.1.1/24)
  • DHCP range for automatic client assignment

4. Admin Password

Set the administrator password for web and CLI access.

5. Configuration Save

The wizard will:

  • Apply network configuration
  • Generate TLS certificates
  • Start all services
  • Save configuration to persistent storage

Post-Setup Access

After setup completes:

  • Web UI: https://<lan-ip> (e.g., https://192.168.1.1)
  • Console CLI: Available on tty1
  • Emergency Console: Available on tty2

Interface Setup

Interface Naming

QuickFW uses standard Linux interface naming:

  • eth0, eth1, ... — Traditional Ethernet names
  • ens33, ens34, ... — Predictable interface names (systemd)
  • enp3s0, enp4s0, ... — PCI-based names

Interface Configuration Files

Network configuration is stored in:

  • /etc/quickfw/appliance.yaml — Main appliance config
  • /etc/quickfw/interfaces.yaml — Interface roles

Role Assignment

Interfaces can be assigned roles:

RolePurpose
wanExternal/internet-facing interface
lanInternal network interface
dmzDemilitarized zone for public services
mgmtOut-of-band management interface

WAN Configuration

DHCP Mode

In DHCP mode, the WAN interface automatically obtains:

  • IP address and subnet mask
  • Default gateway
  • DNS servers

Static Mode

Static configuration requires:

Gateway203.0.113.1
DNS Servers8.8.8.8, 1.1.1.1

PUT /api/interfaces/eth0/config

LAN Configuration

Static IP Assignment

The LAN interface always uses a static IP address:

  • Serves as the default gateway for LAN clients
  • Hosts the web management interface
  • Provides DHCP and DNS services

DHCP & DNS Configuration

DHCP Server (dnsmasq)

QuickFW uses dnsmasq for DHCP services. Configuration includes:

DHCP Options

OptionDescription
RangeStart and end IP for lease pool

GatewayDefault gateway (QuickFW LAN IP)
DNSDNS server (QuickFW LAN IP)

DNS Configuration

Forwarding DNS

QuickFW acts as a DNS forwarder:

  1. Client queries QuickFW DNS
  2. QuickFW checks local overrides
  3. If no match, forwards to upstream DNS
  4. Caches response for future queries

Local DNS Overrides

Create custom DNS entries for internal resources:

POST /api/tools/dns-local
[
  {"hostname": "server.local", "ip": "192.168.1.10"},
  {"hostname": "printer.local", "ip": "192.168.1.20"}
]

Viewing DHCP Leases

GET /api/tools/dhcp-leases

Creating Firewall Rules

Rule Components

FieldDescriptionExamples
NameHuman-readable identifier"Allow HTTP", "Block SSH"
DirectionTraffic flow directioninput, forward, output
ProtocolLayer 4 protocoltcp, udp, icmp, tcp+udp
Source IPSource address/CIDR192.168.1.0/24, any
Source PortSource port(s)80, 443, 1024-65535
Dest IPDestination address/CIDR10.0.0.0/8, any
Dest PortDestination port(s)22, 80, 443
ActionRule actionaccept, drop, reject, log

Rule Direction Reference

Input Chain

Traffic destined for the firewall itself:

  • Management access (HTTPS, SSH)
  • DNS queries to the firewall
  • DHCP requests
  • Ping to firewall interfaces

Forward Chain

Traffic passing through the firewall:

  • LAN to WAN (internet access)
  • WAN to LAN (port forwarded traffic)
  • LAN to LAN (inter-VLAN routing)

Output Chain

Traffic originating from the firewall:

  • DNS queries to upstream servers
  • NTP synchronization
  • Software updates

Example Rules

Allow LAN to WAN (Internet)

{
  "name": "Allow Internet Access",
  "direction": "forward",
  "in_interface": "eth1",
  "out_interface": "eth0",
  "protocol": "any",
  "action": "accept",
  "enabled": true
}

Allow HTTP/HTTPS to Web Server

{
  "name": "Allow Web Traffic",
  "direction": "forward",
  "protocol": "tcp",
  "dst_ip": "192.168.1.10",
  "dst_port": "80,443",
  "action": "accept",
  "enabled": true
}

Block SSH from WAN

{
  "name": "Block External SSH",
  "direction": "input",
  "in_interface": "eth0",
  "protocol": "tcp",
  "dst_port": "22",
  "action": "drop",
  "enabled": true
}

Rule Ordering Best Practices

  1. Management access rules (SSH, HTTPS)
  2. Specific allow rules (particular hosts/services)
  3. General allow rules (LAN to WAN)
  4. Specific block rules (blocked hosts)
  5. General block rules (default deny)

Security Zones

Zone-Based Firewall

Zones simplify rule management by grouping interfaces:

  • WAN Zone — Untrusted external networks
  • LAN Zone — Trusted internal networks
  • DMZ Zone — Semi-trusted public services

Zone Configuration

{
  "zones": [
    {"interface": "eth0", "zone": "wan", "role": "external"},
    {"interface": "eth1", "zone": "lan", "role": "internal"},
    {"interface": "eth2", "zone": "dmz", "role": "public"}
  ]
}

Zone-Based Rules

Use zones instead of interfaces in rules:

{
  "name": "Allow LAN to DMZ HTTP",
  "direction": "forward",
  "src_zone": "lan",
  "dst_zone": "dmz",
  "protocol": "tcp",
  "dst_port": "80,443",
  "action": "accept"
}

NAT & Port Forwarding

Masquerade (Source NAT)

Masquerade hides internal IPs behind the WAN interface IP:

POST /api/nat
{
  "masquerade": [
    {
      "out_interface": "eth0",
      "source_cidr": "192.168.1.0/24"
    }
  ]
}

Port Forwarding (Destination NAT)

Forward external ports to internal hosts:

{
  "port_forward": [
    {
      "protocol": "tcp",
      "dest_port": 443,
      "forward_to": "192.168.1.10:443",
      "in_interface": "eth0"
    },
    {
      "protocol": "tcp",
      "dest_port": 2222,
      "forward_to": "192.168.1.20:22",
      "in_interface": "eth0"
    }
  ]
}

NAT Best Practices

  • Use non-standard external ports for admin services
  • Limit port forwards to specific protocols
  • Document all port forwards for security audits
  • Combine with firewall rules for defense in depth

Static Routes

When to Use Static Routes

  • Small networks without dynamic routing
  • Specific paths to remote networks
  • Default route to upstream gateway
  • Backup routes with higher metrics

Route Configuration

POST /api/routes
{
  "routes": [
    {
      "destination": "10.0.0.0/8",
      "gateway": "192.168.1.254",
      "metric": 100
    },
    {
      "destination": "172.16.0.0/12",
      "gateway": "192.168.1.253",
      "interface": "eth1",
      "metric": 200
    }
  ]
}

Route Priority

Routes are selected based on:

  1. Most specific match (longest prefix)
  2. Lowest metric value
  3. Administrative distance (static vs dynamic)

OSPF Configuration

OSPF Overview

Open Shortest Path First (OSPF) is a link-state routing protocol:

  • Fast convergence
  • Scalable to large networks
  • Supports authentication
  • Uses areas for hierarchical design

Basic OSPF Setup

POST /api/routing/ospf
{
  "enabled": true,
  "router_id": "1.1.1.1",
  "networks": [
    {"prefix": "192.168.1.0/24", "area": 0},
    {"prefix": "10.0.0.0/24", "area": 0}
  ],
  "passive_interfaces": ["eth1"],
  "hello_interval": 10,
  "dead_interval": 40
}

Monitoring OSPF

GET /api/routing/ospf/neighbors
GET /api/routing/table?protocol=ospf

BGP Configuration

BGP Overview

Border Gateway Protocol (BGP) is the routing protocol of the internet:

  • Path-vector protocol
  • Supports policy-based routing
  • Used for multi-homing
  • AS path tracking

Basic BGP Setup

POST /api/routing/bgp
{
  "enabled": true,
  "local_as": 65001,
  "router_id": "1.1.1.1",
  "neighbors": [
    {
      "address": "10.0.0.1",
      "remote_as": 65002,
      "description": "ISP Primary"
    },
    {
      "address": "10.0.0.5",
      "remote_as": 65003,
      "description": "ISP Backup"
    }
  ],
  "networks": ["192.168.0.0/16"]
}

Backup & Restore

Configuration Export

Export all configuration to a single JSON file:

GET /api/config/export

# Save to file
curl -k https://192.168.1.1/api/config/export \
  -b cookies.txt > quickfw-backup-$(date +%Y%m%d).json

Configuration Import

Restore configuration from backup:

POST /api/config/import
{
  "firewall": { ... },
  "nat": { ... },
  "settings": { ... },
  "routes": { ... },
  "roles": { ... }
}

Automatic Backups

QuickFW automatically creates backups before configuration changes:

  • Location: /etc/quickfw/backups/
  • Format: <config>.yaml.<timestamp>.bak
  • Retention: Last 50 backups per config file

Restore from Backup

POST /api/config/restore
{
  "name": "firewall.yaml.1709836800.bak",
  "confirm_password": "your-admin-password"
}

System Updates

Update Process

  1. Export current configuration
  2. Download latest QuickFW ISO
  3. Create new bootable media
  4. Boot from new media
  5. Import saved configuration
Configuration Compatibility

Major version updates may require configuration migration. Always review release notes before updating.

Common Issues

No Internet Access

  1. Check WAN interface has IP: ip addr show eth0
  2. Verify default route: ip route show
  3. Test DNS resolution: nslookup google.com
  4. Check firewall rules aren't blocking
  5. Verify NAT masquerade is configured

Cannot Access Web UI

  1. Verify quickfw-api service is running
  2. Check TLS certificates exist: ls /etc/quickfw/tls.*
  3. Verify port 443 is listening: ss -tlnp | grep 443
  4. Check firewall allows port 443 on input

High CPU Usage

  1. Check active connections: cat /proc/sys/net/netfilter/nf_conntrack_count
  2. Review firewall rule complexity
  3. Check for packet flooding
  4. Review logs for errors
